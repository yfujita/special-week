# FUTURE_WORK — special-week 予測精度改善の引き継ぎ

このドキュメントは、タスク `20260531-yosou-kaizen`（Phase 1〜5）で整備した
バックテスト〜最適化基盤を前提に、**未着手の改善候補**を将来の開発者向けに優先順付きで整理したもの。
着手前に各 Phase の成果（`agent_docs/tasks/20260531-yosou-kaizen/0002〜0009-*.md`）を一読すること。

基本原則（全改善で厳守）:

- **ベースライン非破壊**: `results/` 配下の `baseline_*.json` は過去の各時点の基準値。新規実行は別 `--run-id` で行い、
  これらを上書きしない。最適化前のベースライン値は `baseline_params.yaml` に退避してある。
- **neutral デフォルト**: 新パラメータは「デフォルト＝効果なし」の値で導入し、**追加しても変更前のデフォルト実行結果と
  bit 一致する**ことを確認してから有効化する（リグレッション検出を容易にするため）。
  ※ `ScoringParams` のデフォルト自体は最適化値を反映済みなので、比較基準は「その時点のデフォルト実行結果」。
- **再現性**: 最適化は `seed=42` 固定で再現可能。検証は固定 seed の分割で決定論的に。

## 1. 未活用フィールドで今回見送ったもの

Phase 3（0006）で `RaceResult` 内のフィールドを調査し、データ制約で見送ったもの。

| フィールド | 見送り理由 | 有効化の前提（具体策） | 優先度 |
|---|---|---|---|
| `popularity`（人気） | 全件が **str 型**（ES 設定は integer で型不整合・未解決）。低人気好走の評価はノイズが大きく仮説の妥当性も不確実 | (1) `RaceResult.popularity` を int 化し ES マッピング（`index_settings_horses.json`）と整合させる。(2) その上で「低人気での好走加点」の寄与を Optuna で検証 | 中 |
| `weather`（天候） | 空文字欠損が一定数あり。馬場状態（`course_condition`）と情報が重複。**対象レース当日の天候が race-info に無い** | race-info に当日天候を追加。ただし course_condition と効果が重複しやすいので、馬場状態を先に有効化してから差分効果を見る | 低 |
| `course_condition`（馬場状態） | fixtures 側は取得済みだが、**対象レースの当日馬場状態が race-info / ground_truth に存在せず供給不能**。馬場状態一致ボーナスのフックは neutral で実装済みだが評価できない | **race-info の yml に「当日馬場状態」フィールドを追加**（手入力 or 結果ページから1スクレイピング）→ tipster の `course_condition_match_bonus`（実装済み・現状 neutral 固定）を Optuna 探索対象に復帰させる | 高 |
| `additional_weight`（斤量・per-race） | `RaceResult` に過去戦ごとの斤量フィールドが無く、斤量変化を計算できない | 戦績テーブルの斤量列をパースして `RaceResult` に per-race 斤量を追加（下記 2 と同じ作業） | 中 |
| `run_type`（脚質） | `getRunType()` で算出済みだがスコア未使用。「対象レースのペース傾向」を表すデータが無く照合不能 | ペース予測（展開）モデルが別途必要。難度が高く後回し | 低 |

> 最優先は `course_condition`（馬場状態）。**馬場状態一致ボーナスのロジックは tipster に neutral 同梱済み**なので、
> race-info に当日馬場状態を1フィールド足すだけで有効化・最適化対象化できる（実装コストが最小で効果検証に進める）。

## 3. 汎化改善の次の打ち手（最重要）

0009 の結論：**データ拡充だけでは訓練↔検証の汎化ギャップを頑健には閉じられなかった**
（train は全 seed で改善するが、検証は分割により±に振れ平均ほぼ0）。主因は **モデル側の次元過多**。
よって次の主打ち手はパラメータの次元削減と目的関数の正則化に移す。優先順は以下。

### ① 重賞加点9パラメータの積構造化（最優先）

現状、重賞加点は `grade1_rank1 … grade3_rank5` の **9個の独立パラメータ**で、距離/減点/Phase3 と合わせ
実効十数次元になる。バックテスト母数（top3 は実質 レース数×3 の離散ラベル）に対して次元が多く、訓練分布に張り付きやすい。

- **再設計案**: `grade_base[g1,g2,g3]`（3個）× `rank_multiplier[r1,r3,r5]`（3個）の **積構造**にして
  `加点 = grade_base × rank_mult` で表現する。9 → 6 個（さらに共有すれば 4 個程度）に圧縮できる。
- **副次効果**: 順序制約（`G1≥G2≥G3`、`1着≥3着内≥5着内`）が、`grade_base` と `rank_mult` それぞれの
  単調性だけで自然に保証される（現状の差分積み上げロジックより単純になる）。
- 狙い: 実効次元を十数→数次元に下げ、バックテスト母数に対する過適合を緩和する。

### ② Optuna 目的関数の k-fold CV 化（正則化）

- 現状の目的は「全レースで1回バックテストした複合スコア」。これを **k-fold CV の平均スコア**に変えると、
  特定の分割に過適合したパラメータが選ばれにくくなる（正則化）。
- `holdout_validate.py` に k-fold ロジックがあるので、`optimize.py` の目的関数を CV 平均に差し替える形で実装できる。
- 狙い: 訓練スコアだけを追わず、汎化スコアそのものを最適化する。

### ③ さらなるレース拡充

- `build_race_manifest.py` の `TARGET_KAISAI_DATES` に開催日を足すだけで対象レースを追記できる。
- レース数を増やすほど検証側の分解能（1頭の3着内入替で動く平均値 ≈ 1/レース数）が細かくなり、
  小さな真の改善が分割ノイズに埋もれにくくなる。
- ①②と**併用**するのが効果的（単独のデータ拡充は 0009 で効果限定的と判明済み）。

**優先順**: ①（次元削減）→ ②（CV 目的化）→ ③（データ拡充）。①②でモデル側の過適合を抑え、③で
検証分解能を上げる、という順序が費用対効果が高い。

## 4. 着手手順（どのファイルをどう変えるか）

### 共通の流れ

1. **変更前のデフォルト実行結果を控える**: `python3 run_backtest.py --run-id before_change` でデフォルト実行し、
   その指標値を記録しておく（後でパラメータ追加後と比較し、neutral 担保＝挙動不変を確認するため）。
2. **パラメータ追加/変更**: `scoring_params.py` の `ScoringParams` にフィールドを追加（**デフォルト＝neutral**）。
   `tipster.py` の `_get_win_score()` にガード節付きで組み込む（neutral 時は演算が増えないように）。
3. **ベースライン再確認**: 再度デフォルト実行し、ベースラインと bit 一致することを確認（リグレッション検出）。
4. **最適化**: `optimize.py` の探索範囲に新パラメータを追加し、`python3 optimize.py --n-trials 200 --out <new>.yaml`。
5. **汎化検証のやり直し**: `python3 holdout_validate.py --n-trials 200 --kfold 3` で train/val の改善を確認。
   検証側がベースラインを超えるか（過学習していないか）を見る。
6. **結果保存は別 run-id で**: `run_backtest.py --params <new>.yaml --run-id <id>` → `results/phase3_<id>.json`。
   **ベースライン保護ファイルは絶対に上書きしない**。

### 改善別の主な変更ファイル

| 改善 | 主に変更するファイル |
|---|---|
| 馬場状態一致ボーナスの有効化（1-③） | `race-info/*.yml`（当日馬場状態追加）→ `special-week.py` / `run_backtest.py` で race_info に渡す → `optimize.py` の探索に `course_condition_match_bonus` 復帰 |
| 新規フィールド収集（2） | `test_derby2026.py` の戦績パース（後方互換注意）→ `horse.py`（`RaceResult` 列追加）→ `collect_fixtures.py`（再収集）→ `scoring_params.py` / `tipster.py`（neutral 追加）→ `optimize.py`（探索追加） |
| 重賞加点の積構造化（3-①） | `scoring_params.py`（`grade_base` / `rank_multiplier` へ再設計）→ `tipster.py`（加点計算）→ `optimize.py`（探索・順序制約を単調性で表現） |
| 目的関数の CV 化（3-②） | `optimize.py`（目的関数を k-fold CV 平均に変更。`holdout_validate.py` の分割ロジックを流用） |
| レース拡充（3-③） | `build_race_manifest.py`（`TARGET_KAISAI_DATES` 追記）→ `collect_ground_truth.py` / `collect_fixtures.py`（新規分のみ取得）→ `run_backtest.py --run-id <new>` |
